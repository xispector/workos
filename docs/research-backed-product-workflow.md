# Research-Backed Product Workflow Decision

작성일: 2026-07-05

목적: 개인 Work OS를 실제 개인용 웹 앱 MVP로 만들기 위해, 기존 일정/태스크/프로젝트 관리 앱의 UI/UX 패턴을 조사하고 이 제품에 맞는 기본 화면 구성과 사용 워크플로우를 결정한다.

이 문서는 구현 문서가 아니다. Google AI Studio 또는 이후 제품 설계 단계에 넣을 수 있는 제품 방향 결정서다.

## 1. 조사한 앱과 핵심 패턴

### Todoist

Todoist는 빠른 수집과 Today/Upcoming/Filter 중심 구조가 강하다. 공식 문서에서 Quick Add는 자연어로 날짜, 마감, 라벨 등을 입력할 수 있고, 필터는 프로젝트, 라벨, 우선순위, 날짜 등으로 작업을 좁혀 보는 구조다.

참고:

- https://www.todoist.com/help/articles/use-task-quick-add-in-todoist-va4Lhpzz
- https://www.todoist.com/help/articles/introduction-to-filters-V98wIH
- https://www.todoist.com/

빌릴 점:

- 빠른 수집은 매우 중요하다.
- 사용자가 모든 필드를 직접 채우지 않아도 자연어로 입력할 수 있어야 한다.
- Today 같은 실행 중심 뷰가 필요하다.

주의할 점:

- 필터/라벨을 사용자가 직접 설계하게 하면 부담이 커질 수 있다.
- 이 제품에서는 태그와 필터를 AI가 제안해야 한다.

### Things

Things는 Inbox, Today, Upcoming, Anytime, Someday처럼 할 일을 "언제 볼 것인가" 기준으로 나눈다. 공식 문서는 프로젝트/영역 같은 맥락 관점과 Today/Upcoming 같은 시간 관점을 함께 제공한다고 설명한다.

참고:

- https://culturedcode.com/things/support/articles/4001304/
- https://culturedcode.com/things/support/articles/2803579/

빌릴 점:

- 모든 할 일이 항상 보이면 부담스럽다.
- 지금 볼 것, 나중에 볼 것, 언젠가 볼 것을 분리해야 한다.
- 보류는 실패가 아니라 "지금 보지 않아도 되는 상태"로 설계해야 한다.

주의할 점:

- Things의 단순함은 좋지만, 이 제품은 AI가 프로젝트 맥락을 이해해야 하므로 프로젝트 상태 문서가 더 필요하다.

### Sunsama

Sunsama는 Guided Planning, Weekly Objectives, Daily Planning, Daily Shutdown/Review 흐름이 강하다. 주간 목표를 먼저 잡고, 일일 계획에서 작업을 끌어와 시간 용량과 비교한다.

참고:

- https://help.sunsama.com/docs/usage-guides/daily-planning/
- https://help.sunsama.com/docs/usage-guides/weekly-objectives/weekly-planning/
- https://help.sunsama.com/docs/usage-guides/weekly-objectives/weekly-review/
- https://www.sunsama.com/features/guided-planning-and-reviews

빌릴 점:

- 주간 판단과 일일 실행을 분리해야 한다.
- 이번 주 목표/전진 프로젝트가 있어야 오늘 작업 추천이 흔들리지 않는다.
- 하루 끝에는 짧은 종료/로그 루틴이 있으면 좋다.

주의할 점:

- 매일 긴 계획 의식을 강제하면 부담이 된다.
- 이 제품은 "가이드된 계획"보다 "AI가 맥락을 줄여주는 실행 진입"이 더 중요하다.

### Akiflow

Akiflow는 Universal Inbox, Command Bar, Calendar/Time Blocking이 강하다. 여러 도구에서 들어온 일을 하나의 Inbox로 모으고, Command Bar로 빠르게 캡처하며, 작업을 캘린더에 배치한다.

참고:

- https://akiflow.com/
- https://product.akiflow.com/help/articles/5284502-your-inbox
- https://product.akiflow.com/help/articles/6483573-command-bar
- https://product.akiflow.com/help/articles/9351674-how-to-create-tasks
- https://product.akiflow.com/help/articles/3677363-time-blocking-101

빌릴 점:

- 수집은 항상 빠르게 열려야 한다.
- 작업은 여러 소스에서 들어올 수 있으므로 Inbox 개념이 필요하다.
- 실행 가능한 작업은 캘린더/시간과 연결될 수 있다.

주의할 점:

- 처음 MVP에서 시간 블로킹을 중심에 두면 복잡해진다.
- 사용자는 목표일과 예상 시간을 정하기 어려워하므로, 처음부터 캘린더 배치 중심으로 가면 부담이 생길 수 있다.

### Motion

Motion은 AI가 작업의 우선순위, 마감, 가능 시간을 고려해 자동으로 캘린더에 배치하는 접근이다.

참고:

- https://www.usemotion.com/
- https://www.usemotion.com/help/time-management/auto-scheduling

빌릴 점:

- AI가 우선순위와 일정 조정을 도와야 한다.
- 일정이 바뀌면 계획도 다시 조정되어야 한다.

주의할 점:

- 완전 자동 스케줄링은 신뢰 문제가 생길 수 있다.
- 이 제품은 "AI가 알아서 결정"보다 "AI가 근거 있는 제안을 하고 사용자가 조정"하는 방식이 맞다.

### OmniFocus

OmniFocus는 Inbox, Projects, Tags, Forecast, Review, Perspectives 구조가 강하다. 특히 Review는 데드라인 없는 프로젝트를 주기적으로 다시 보게 하는 데 유용하다.

참고:

- https://support.omnigroup.com/documentation/omnifocus/universal/4.8.11/en/perspectives/
- https://inside.omnifocus.com/james-dempsey

빌릴 점:

- 프로젝트별 정기 리뷰가 필요하다.
- 데드라인 없는 프로젝트는 "다음 확인일"이나 리뷰 장치가 없으면 방치된다.
- 사용자 맞춤 Perspective, 즉 특정 조건에 맞는 작업 보기 개념이 유용하다.

주의할 점:

- Perspectives/Tags를 사용자가 직접 많이 설계하는 방식은 이 제품에 맞지 않는다.
- AI가 개인화된 Perspective를 만들어주는 방향이 더 적합하다.

### Notion / Linear

Notion은 프로젝트, 태스크, 문서가 함께 있는 구조가 강하다. Notion 공식 문서는 프로젝트와 태스크가 노트/문서와 함께 관리될 수 있다고 설명하고, Task database와 My tasks/Sprints 같은 구조를 제공한다.

Linear는 프로젝트/이니셔티브 업데이트, 이슈, 프로젝트 상태를 다루는 데 강하고, 프로젝트 업데이트는 진행 상태와 맥락을 정리하는 방식에 가깝다.

참고:

- https://www.notion.com/help/guides/getting-started-with-projects-and-tasks
- https://www.notion.com/help/sprints
- https://www.notion.com/product/projects
- https://linear.app/docs/initiative-and-project-updates

빌릴 점:

- 프로젝트와 문서는 분리되면 안 된다.
- 프로젝트 상태 문서가 있어야 AI가 태스크를 제대로 쪼갤 수 있다.
- 최근 진척 업데이트는 프로젝트 복귀에 중요하다.

주의할 점:

- Notion식 자유도는 강하지만, 구조를 직접 설계해야 하는 부담이 크다.
- 이 제품은 자유로운 문서보다 "AI가 유지해주는 프로젝트 이해 카드"가 필요하다.

## 2. 조사에서 얻은 공통 UI/UX 원칙

### 원칙 1. 빠른 수집은 거의 모든 좋은 생산성 앱의 공통 기반이다

Todoist, Akiflow, Things 모두 Inbox/Quick Add/Command Bar 계열의 빠른 수집 구조를 갖고 있다.

결정:

이 제품도 빠른 수집은 핵심이다. 단, 사용자가 태그/우선순위/예상 시간까지 직접 입력하게 하지 않는다. 자유 입력 후 AI가 구조화한다.

### 원칙 2. 첫 화면은 "전체 데이터"가 아니라 "지금 볼 것"을 보여줘야 한다

Todoist의 Today, Things의 Today, OmniFocus의 Forecast, Sunsama의 Daily Planning은 모두 전체 목록이 아니라 오늘/지금 처리할 것을 좁혀 보여준다.

결정:

첫 화면은 전체 프로젝트 대시보드가 아니라 "오늘 작업 화면"이어야 한다.

### 원칙 3. 주간 판단과 일일 실행은 분리되어야 한다

Sunsama의 Weekly Objectives와 Daily Planning, Notion/Linear의 프로젝트/스프린트 구조는 큰 목표와 오늘 작업을 분리한다.

결정:

주간 판단 시스템은 별도로 두고, 첫 화면은 그 판단 결과를 실행 가능한 작업으로 보여준다.

### 원칙 4. 프로젝트 맥락이 없으면 태스크 추천은 얕아진다

Notion/Linear는 프로젝트와 문서를 함께 둔다. OmniFocus도 프로젝트와 Review를 통해 데드라인 없는 작업을 다시 본다.

결정:

이 제품의 AI는 반드시 프로젝트 목표, 결과물 목록, 최근 진척 요약, 다음 작은 결과물을 알아야 한다.

### 원칙 5. 완전 자동화보다 근거 있는 제안이 더 안전하다

Motion은 자동 스케줄링의 강점을 보여주지만, 개인 맥락이 충분하지 않으면 신뢰 문제가 생길 수 있다.

결정:

AI는 우선순위와 목표일을 제안하되, 근거를 보여주고 사용자가 조정할 수 있어야 한다.

## 3. 이 제품에 맞는 기본 화면 결정

### 첫 화면: 오늘 작업 화면

첫 화면은 "빠른 수집 화면"도 아니고 "프로젝트 대시보드"도 아니다.

결정:

첫 화면은 오늘 작업 화면이다.

이유:

- 사용자의 핵심 문제는 "뭐부터 해야 할지 모르겠다"이다.
- 앱을 열었을 때 해야 할 작업이 바로 보여야 한다.
- 단순 전체 목록이 아니라 AI가 현재 맥락에 맞게 좁힌 작업이어야 한다.

### 첫 화면 기본 구성

1. 상단 실행 컨텍스트 바
2. 오늘 추천 작업 목록
3. 빠른 수집 입력
4. 이번 주 전진 프로젝트 요약

### 1. 상단 실행 컨텍스트 바

예:

`집 · 노트북 · 조용함 · 보통`

역할:

- 지금 추천이 어떤 조건을 기준으로 나오는지 보여준다.
- 사용자가 환경이나 작업 강도를 즉시 바꿀 수 있다.

필드:

- 장소: 집 / 외부 / 이동 중
- 도구: 노트북 가능 / 휴대폰만
- 환경: 조용함 / 산만함
- 작업 강도: 가볍게 / 보통 / 깊게

### 2. 오늘 추천 작업 목록

첫 화면의 중심이다.

기본은 3개 추천으로 시작한다.

각 작업 카드에는 다음 정보가 있어야 한다.

- 작업 제목
- 연결된 프로젝트
- 예상 작업 강도
- 추천 이유 한 줄
- 시작 버튼

예:

작업: 수집 시스템 입력 흐름 사례 3개 적기  
프로젝트: Personal Work OS  
이유: 이번 주 전진 프로젝트이고, 현재 `노트북 · 조용함 · 보통`에서 바로 정리 가능함.

### 3. 빠른 수집 입력

첫 화면에서 바로 접근 가능해야 하지만, 화면 최상단 중심 요소는 아니다.

역할:

- 떠오른 해야 할 일, 생각, 로그, 아이디어를 바로 넣는다.
- 입력 후 AI가 프로젝트 연결과 태스크 분해를 시도한다.

### 4. 이번 주 전진 프로젝트 요약

전체 프로젝트를 다 보여주지 않는다.

보여줄 것:

- 이번 주 전진 프로젝트 1~2개
- 각 프로젝트의 다음 작은 결과물
- 최근 진척 한 문장

역할:

- 추천 작업이 어떤 큰 흐름에서 나왔는지 보여준다.
- 전체 대시보드로 빠지는 선택 피로를 줄인다.

## 4. 추천 작업 정렬 방식 결정

사용자가 아직 시스템을 써본 적이 없으므로 정렬 기준을 직접 고르게 하지 않는다.

결정:

AI가 다음 순서로 점수를 매겨 정렬한다.

1. 이번 주 전진 프로젝트와 연결되는가
2. 현재 실행 컨텍스트에서 가능한가
3. 다음 작은 결과물에 가까워지는가
4. 심리적 부담이 낮거나 작업 강도와 맞는가
5. 지연 위험이나 목표일 압박이 있는가
6. 최근 로그가 끊겨 복귀가 필요한가

이 정렬 이유는 각 추천 카드에 한 줄로 표시한다.

## 5. 기본 사용 워크플로우

### 매일 앱을 열 때

1. 오늘 작업 화면을 본다.
2. 상단 실행 컨텍스트가 현재 상황과 맞는지 확인한다.
3. 추천 작업 3개를 본다.
4. 하나를 시작한다.
5. 작업이 끝나면 한 줄 로그를 남기거나 스킵한다.

### 해야 할 일이 떠오를 때

1. 빠른 수집에 자유 입력한다.
2. AI가 프로젝트 연결을 시도한다.
3. 불확실하면 좁은 질문 하나를 한다.
4. 프로젝트 맥락을 불러와 태스크를 쪼갠다.
5. 현재 컨텍스트에서 가능한 작업으로 변환한다.
6. 추천 후보에 추가한다.

### 흐름이 깨졌을 때

1. 상태 전환 진입을 누른다.
2. 5분 유산소 복귀 행동을 한다.
3. 오늘 작업 화면으로 돌아온다.
4. 작업 강도를 `가볍게`로 낮춘다.
5. 가벼운 추천 작업 3개 중 하나를 고른다.

### 주간 계획 때

1. 프로젝트별 결과물 목록과 최근 진척 요약을 본다.
2. AI가 전진/유지/보류/막힘을 제안한다.
3. 각 제안의 근거를 확인한다.
4. 이번 주 전진 프로젝트 1~2개를 확정한다.
5. 각 전진 프로젝트의 다음 작은 결과물과 목표일을 확정한다.

### 작업 완료 후

1. 완료 버튼을 누른다.
2. 시스템이 "한 줄로 남길래?"라고 묻는다.
3. 텍스트/음성으로 남기거나 스킵한다.
4. 남기면 AI가 높은 신뢰도 로그로 정리한다.
5. 스킵하면 낮은 신뢰도 자동 로그를 남긴다.

## 6. MVP에서 피해야 할 것

### 피해야 할 것 1. 첫 화면을 프로젝트 대시보드로 만드는 것

프로젝트 대시보드는 전체 맥락에는 좋지만, 사용자의 핵심 문제인 "지금 뭐 하지?"를 바로 해결하지 못할 수 있다.

### 피해야 할 것 2. 처음부터 캘린더/시간 블로킹 중심으로 만드는 것

사용자는 예상 시간과 이상적 데드라인을 정하기 어렵다고 했다. 따라서 MVP는 시간 블로킹보다 실행 가능한 작업 추천이 먼저다.

### 피해야 할 것 3. 태그/우선순위/예상 시간을 사용자가 직접 입력하게 하는 것

이 제품의 차별점은 암묵지를 AI가 구조화하는 것이다. 수동 메타데이터 입력이 많으면 기존 태스크 앱과 같은 문제가 반복된다.

### 피해야 할 것 4. AI가 자동으로 모든 것을 확정하는 것

AI 추천은 근거를 보여줘야 하고, 중요한 판단은 사용자가 수정 가능해야 한다.

### 피해야 할 것 5. 완료 후 긴 회고 질문을 매번 하는 것

작업 완료 후에는 한 줄 로그만 기본으로 요청하고, 스킵 가능해야 한다.

## 7. 결론: 이 제품의 기본 UX 방향

이 제품은 Todoist처럼 빠르게 수집하고, Sunsama처럼 주간 목표와 일일 실행을 분리하고, Akiflow처럼 작업을 한곳에 모으되, Motion처럼 완전 자동 스케줄링으로 가지 않는다. Notion/Linear처럼 프로젝트 문서와 맥락을 중요하게 보지만, 사용자가 직접 복잡한 데이터베이스를 설계하게 하지 않는다.

핵심 차별점:

사용자는 자유롭게 입력하고, AI가 프로젝트 맥락을 바탕으로 실행 가능한 작업으로 바꾼다.

MVP의 첫 화면:

오늘 작업 화면.

가장 중요한 첫 화면 요소:

AI가 현재 컨텍스트 기준으로 정리한 해야 할 작업 3개.

기본 워크플로우:

수집 -> 프로젝트 연결 -> 태스크 분해 -> 현재 환경에 맞게 변환 -> 추천 -> 실행 -> 로그 -> 프로젝트 이해 업데이트.

이 방향이 현재 사용자의 문제에 가장 잘 맞는다.
