# Product Brief: Personal Work OS

작성일: 2026-07-05

## 1. 제품 한 줄 설명

Personal Work OS는 개인의 할 일, 프로젝트 맥락, 실행 가능한 다음 행동, 작업 로그를 AI가 연결해주는 개인용 웹 앱이다.

목표는 단순한 투두 리스트가 아니라, 사용자가 "지금 무엇을 해야 하는지"를 덜 고민하고 실제 결과물로 이어지게 돕는 것이다.

## 2. 대상 사용자

초기 대상 사용자는 1명이다.

- 창업, 개인 개발, 공부, SNS, 네트워킹처럼 자기주도적 일을 병렬로 운영한다.
- 할 일은 많지만 우선순위와 다음 행동이 자주 흐려진다.
- 외부 일정이나 컨디션 저하로 흐름이 깨지면 프로젝트 복귀가 어렵다.
- 메모와 로그를 남기고 싶지만 직접 정리하는 부담이 크다.
- AI가 자신의 맥락을 이해하고 태스크 분해, 추천, 목표 조정을 도와주길 원한다.

초기 MVP는 다사용자 SaaS가 아니다. 개인 사용에 최적화한다.

## 3. 해결할 핵심 문제

### 문제 1. 해야 할 것은 있지만 지금 무엇부터 해야 할지 모른다

여러 프로젝트와 생활 업무가 병렬로 존재한다. 모든 일이 중요해 보여 선택 피로가 생긴다.

### 문제 2. 프로젝트 흐름이 끊기면 복귀가 어렵다

밋업, 외부 일정, 컨디션 저하 이후 "어디까지 했는지", "다음에 뭘 해야 하는지"가 흐려진다.

### 문제 3. 태스크가 너무 크거나 애매하다

창업, 개발, 공부처럼 자기주도적 일은 "무엇을 할지" 자체를 설계해야 한다. 실행보다 정의 부담이 먼저 온다.

### 문제 4. 기록은 필요하지만 기록 자체가 부담이다

프로젝트 맥락을 쌓으려면 로그가 필요하다. 하지만 매번 정돈된 기록을 직접 쓰면 지속하기 어렵다.

### 문제 5. 목표일과 우선순위를 직접 판단하기 어렵다

경험, 컨디션, 환경, 다른 업무를 모두 고려해 현실적인 목표일을 정하기 어렵다.

## 4. 제품 원칙

### 원칙 1. 입력은 자유롭게, 구조화는 AI가 한다

사용자는 해야 할 일, 생각, 로그를 자연어 또는 음성으로 입력한다. AI가 프로젝트 연결, 태스크 분해, 태그/크기/우선순위 후보를 제안한다.

### 원칙 2. 전체 목록보다 지금 가능한 작업을 우선한다

첫 화면은 전체 프로젝트 대시보드가 아니라 오늘 작업 화면이다. 사용자가 앱을 열었을 때 가장 먼저 해야 할 작업 후보가 보여야 한다.

### 원칙 3. 프로젝트 맥락이 태스크 판단의 기준이다

AI는 단편적인 할 일만 보고 판단하지 않는다. 프로젝트 목표, 결과물 목록, 최근 진척, 다음 작은 결과물을 바탕으로 태스크를 쪼개고 추천한다.

### 원칙 4. AI는 자동 결정자가 아니라 근거 있는 보조자다

AI는 추천과 판단을 제안하되, 항상 짧은 이유를 보여준다. 중요한 판단은 사용자가 수정할 수 있다.

### 원칙 5. 기록은 기본 요청하되 강제하지 않는다

작업 완료 후 한 줄 로그를 요청한다. 사용자가 스킵하면 낮은 신뢰도의 자동 완료 로그만 남긴다.

### 원칙 6. 실패보다 재조정이 기본이다

목표일이 늦어지면 실패로 보지 않는다. AI가 목표일 조정 이유를 기록하고 새 목표일을 제안한다.

## 5. 이론과 레퍼런스 기반 근거

### GTD

Getting Things Done은 capture, clarify, organize, reflect, engage 흐름을 강조한다. 이 제품은 이 흐름을 AI 기반으로 재해석한다.

- Capture: 빠른 수집
- Clarify: AI의 프로젝트 연결과 맥락 질문
- Organize: 프로젝트 이해 카드와 작업 후보 정리
- Reflect: 주간 판단과 프로젝트 리뷰
- Engage: 오늘 작업 화면에서 실행

참고: https://gettingthingsdone.com/what-is-gtd/

### 구현의도 이론

Implementation intention은 특정 상황에서 어떤 행동을 할지 미리 연결하는 방식이다. 이 제품의 실행 컨텍스트 바와 10분 행동 추천은 "현재 환경이면 이 행동"을 만들기 위한 장치다.

참고:

- https://cancercontrol.cancer.gov/brp/research/constructs/implementation-intentions
- https://sparq.stanford.edu/sites/g/files/sbiybj19021/files/media/file/gollwitzer_brandstatter_1997_-_implementation_intentions_effective_goal_pursuit.pdf

### 인지부하 이론

작업 선택, 분류, 우선순위 판단, 예상 시간 입력은 모두 작업 전 인지부하를 만든다. 이 제품은 필드 입력을 줄이고 AI가 구조화를 맡아 작업 시작 전 부하를 낮춘다.

참고:

- https://www.sciencedirect.com/science/article/pii/0959475294900035
- https://www.uky.edu/~gmswan3/544/Cognitive_Load_%26_ID.pdf

### 미완료 과업과 주의 점유

미완료 과업은 반복적인 생각과 회복 방해로 이어질 수 있다. 이 제품은 미완료 과업을 머릿속에 두지 않고 프로젝트 상태, 다음 행동, 목표일 조정으로 외부화한다.

참고:

- https://pubmed.ncbi.nlm.nih.gov/41554526/
- https://www.researchgate.net/publication/301562131_Zeigarnik%27s_Sleepless_Nights_How_Unfinished_Tasks_at_the_End_of_the_Week_Impair_Employee_Sleep_on_the_Weekend_Through_Rumination

## 6. MVP 성공 기준

MVP가 성공했다고 볼 수 있는 기준:

- 사용자가 해야 할 일을 빠르게 입력할 수 있다.
- 입력된 할 일이 적절한 프로젝트에 연결된다.
- AI가 큰 태스크를 현재 환경에서 가능한 작은 행동으로 바꾼다.
- 첫 화면에서 오늘 할 작업 후보 3개를 볼 수 있다.
- 각 추천 작업에 이유가 붙는다.
- 작업 완료 후 로그가 프로젝트 맥락에 쌓인다.
- 로그가 없어도 낮은 신뢰도의 최소 기록이 남는다.
- 프로젝트를 다시 열었을 때 목표, 결과물, 최근 진척, 다음 행동이 보인다.

## 7. MVP에서 제외할 것

초기 버전에서 제외한다.

- 회원가입/팀/공유 기능
- 결제
- 자동 위치 감지
- 완전 자동 캘린더 스케줄링
- 화면 녹화 기반 자동 맥락 수집
- 복잡한 태그/필터 빌더
- 모든 프로젝트의 완전 자동 요약
- 정교한 통계 기반 개인화 모델

## 8. 제품의 핵심 차별점

기존 투두 앱은 할 일을 저장한다.

이 제품은 할 일을 프로젝트 맥락에 연결하고, 지금 실행 가능한 조각으로 바꾸며, 작업 후 로그를 통해 다시 프로젝트 이해를 업데이트한다.

핵심 루프:

수집 -> 프로젝트 연결 -> 태스크 분해 -> 현재 환경에 맞게 변환 -> 추천 -> 실행 -> 로그 -> 프로젝트 이해 업데이트
