# Personal Work OS Architecture Notes

작성일: 2026-07-05

이 문서는 개인 Work OS를 설계하기 위해 지금까지 인터뷰하며 정리한 핵심 문제, 임시 채택한 시스템, 각 시스템이 관리해야 할 정보, 그리고 MVP 방향을 정리한 것이다.

아직 데이터베이스 설계나 구현 설계가 아니다. 목적은 큰 구조와 흐름을 이해하고, 이후 웹 앱으로 만들기 위한 기준을 잡는 것이다.

## 1. 핵심 문제 인식

현재 가장 큰 문제는 단순히 할 일이 많다는 것이 아니다.

해야 할 것들은 있지만, 여러 일이 병렬로 놓여 있고 무엇부터 해야 할지 자주 흐려진다. 밋업이나 외부 일정 이후 생활 패턴이 깨지면 하던 일의 흐름을 놓치고, 프로젝트에 다시 복귀하기 어렵다. 컨디션이 낮을 때는 작은 일도 시작하기 싫고, 결국 유튜브를 보며 멍한 상태로 빠지는 경우가 있다.

또한 많은 일이 창업, 개인 개발, 공부, SNS, 네트워킹처럼 자기주도적이다. 이런 일들은 단순 체크리스트가 아니라, 무엇을 왜 어떤 순서로 할지 직접 설계해야 한다. 그래서 실행보다 먼저 일을 정의하는 부담이 크다.

가장 크게 느끼는 부담은 "결과물을 못 내고 있다"는 감각이다. 따라서 Work OS는 시간을 기록하는 도구보다, 프로젝트가 실제 결과물로 이어지도록 돕는 시스템이어야 한다.

## 2. 현재 핵심 시스템 5개

지금까지 논의한 기능들을 너무 잘게 나누지 않고, 다음 5개 핵심 시스템으로 묶는다.

1. 수집 시스템
2. 프로젝트 이해 시스템
3. 주간 판단 시스템
4. 실행 추천 시스템
5. 계획 조정 시스템

이 5개는 서로 다른 문제를 해결한다.

## 3. 수집 시스템

### 해결하는 문제

머릿속에 있는 해야 할 일, 아이디어, 메모, 고민, 작업 로그가 흩어지는 문제를 해결한다.

가장 먼저 필요한 시스템으로 판단했다. 데이터가 쌓여야 프로젝트 이해, 우선순위 판단, 태스크 분해, 실행 추천, 계획 조정이 가능하기 때문이다.

### 중요한 설계 원칙

수집은 노동집약적이면 안 된다.

사용자가 우선순위, 예상 시간, 태그, 프로젝트 분류를 매번 직접 정하게 하면 시스템은 금방 무너진다. 특히 많은 일에는 암묵지가 있어서 처음 입력할 때 분류가 명확하지 않다.

따라서 수집 시스템의 기본 원칙은 다음과 같다.

- 사용자는 자유롭게 입력한다.
- 텍스트와 음성 입력을 모두 허용한다.
- AI가 먼저 프로젝트 연결과 분류를 시도한다.
- AI가 확신이 낮을 때만 질문한다.
- 질문은 한 번에 하나만 한다.
- 선택지를 너무 많이 보여주지 않는다.
- 태그, 크기, 우선순위, 예상 시간은 AI가 제안한다.
- 사용자는 틀렸을 때만 수정한다.

### 수집 후 처리 흐름

해야 할 일이 들어오면 AI는 먼저 프로젝트 연결을 시도한다.

흐름:

1. 자유 입력
2. 프로젝트 연결 시도
3. 연결이 불확실하면 좁은 맥락 질문 하나
4. 프로젝트 목표와 현재 상태 불러오기
5. 큰 태스크라면 분해
6. 현재 실행 컨텍스트에 맞게 가능한 작업으로 변환
7. 실행 추천 후보로 넘김

### 맥락 질문 원칙

AI 질문은 분류표처럼 느껴지면 안 된다.

좋은 질문:

- "이건 기존 프로젝트를 이어가는 일이야?"
- "이 일은 지금 만들고 있는 앱 프로젝트와 관련 있어?"
- "이건 결과물을 만들기 위한 작업이야?"

나쁜 질문:

- "다음 8개 카테고리 중 골라줘."
- "우선순위, 마감일, 예상 시간, 태그를 입력해줘."

### 태스크 분해 원칙

태스크 분해에서 가장 중요한 것은 전체 순서를 완벽히 잡는 것이 아니라, 지금 환경에서 할 수 있는 작업으로 바꾸는 것이다.

예를 들어 같은 태스크라도:

- 집 + 노트북 + 조용함: 문서 구조 작성
- 이동 중 + 휴대폰 + 산만함: 음성으로 핵심 생각 1개 남기기
- 외부 + 노트북 + 사람 많음: 기존 메모 읽고 표시하기

처럼 달라져야 한다.

## 4. 프로젝트 이해 시스템

### 해결하는 문제

AI가 우선순위를 정하고 태스크를 잘게 쪼개려면, 프로젝트 자체에 대한 이해가 필요하다.

단편적인 할 일만으로는 부족하다. AI가 프로젝트의 목표, 현재 상태, 결과물, 최근 진척을 알아야 제대로 판단할 수 있다.

### 프로젝트 이해의 핵심 정보

프로젝트를 처음 이해할 때 가장 중요한 정보는 두 가지다.

- 목표
- 현재 어디까지 진행했는지

목표는 "최종적으로 만들고 싶은 결과물" 중심으로 적는 것이 자연스럽다.

예:

- Work OS 웹 앱 MVP 만들기
- AI 로그 정리 기능 프로토타입 만들기
- 밋업 후속 연락 관리 흐름 만들기

현재 상태는 "지금까지 만든 것"을 먼저 본다. 여기서 만든 것은 완성된 앱이나 파일만 뜻하지 않는다. 프로젝트를 다음 단계로 이어갈 수 있게 해주는 남은 흔적 전체를 의미한다.

### 지금까지 만든 것

현재 상태를 보여줄 때는 다음 두 가지가 중요하다.

1. 결과물 목록
2. 최근 진척 요약

결과물 목록은 실제로 남은 화면, 문서, 코드, 초안 같은 것이다.

최근 진척 요약은 최근 며칠 또는 최근 작업에서 무엇이 진행됐는지 AI가 1~3문장으로 요약한 것이다.

### 프로젝트 상태 카드

프로젝트마다 하나의 상태 카드가 필요하다.

초기 항목:

- 최종 목표
- 결과물 목록
- 최근 진척 요약
- 상태 단계 + 현재 맥락 한 문장
- 다음 작은 결과물
- 다음 10분 행동
- 막힌 점
- 최근 작업 로그

상태 표현은 단계와 문장을 함께 쓴다.

예:

`실행 중` - 수집 시스템의 입력/분류 흐름을 정리했고, 프로젝트 연결과 태스크 분해 규칙을 다듬는 중.

상태 단계 후보:

- 대기
- 조사 중
- 설계 중
- 실행 중
- 막힘
- 완료

## 5. 주간 판단 시스템

### 해결하는 문제

모든 일이 다 중요해 보여서 무엇을 밀고 무엇을 내려놓을지 모르는 문제를 해결한다.

중요도는 단순히 "A가 B보다 중요하다"가 아니다. 이번 주에 무엇을 전진시키고, 무엇은 유지하고, 무엇은 보류해도 되는지를 판단해야 한다.

### 주간 역할

프로젝트나 영역은 이번 주 기준으로 다음 역할 중 하나를 가질 수 있다.

- 전진: 이번 주 실제 결과물을 남길 것
- 유지: 완전히 놓치지 않기 위해 작은 행동만 할 것
- 보류: 이번 주에는 의도적으로 안 할 것
- 막힘: 실행보다 먼저 결정이나 정리가 필요한 것

### 판단 근거

AI가 판단을 제안할 때는 반드시 근거를 보여줘야 한다.

특히 보류 제안은 신뢰가 중요하다. 보류해도 괜찮다고 느끼려면 다음 근거가 필요하다.

- 최근에 이미 충분히 진행됐는가
- 나중에 해도 목표일이나 적절한 진행 속도에서 너무 처지지 않는가
- 다음 행동이 남아 있어 복귀 가능한가

보류는 실패가 아니라 리듬 조절로 느껴져야 한다.

## 6. 실행 추천 시스템

### 해결하는 문제

지금 뭘 해야 할지 모르거나, 중요한 일이 너무 커 보여서 시작하지 못하는 문제를 해결한다.

실행 추천은 "중요한 일"만 보여주면 안 된다. 중요한 흐름에서 나온, 지금 환경과 작업 강도에서 가능한 작업을 보여줘야 한다.

### 상단 실행 컨텍스트 바

실행 추천의 기준은 화면 상단에서 항상 보여야 한다.

예:

`집 · 노트북 · 조용함 · 가볍게`

구성:

- 장소: 집 / 외부 / 이동 중
- 도구: 노트북 가능 / 휴대폰만
- 환경: 조용함 / 산만함
- 작업 강도: 가볍게 / 보통 / 깊게

환경과 작업 강도는 사용자가 직접 바꿀 수 있어야 한다. 특히 환경은 시스템이 매번 묻기보다 사용자가 바꾸고 싶을 때 직접 바꾸는 방식이 좋다.

### 작업 강도

작업 강도는 에너지 기록이라기보다 추천 난이도를 조절하는 손잡이다.

- 가볍게: 단발성, 낮은 판단 부담, 후속 업무가 적고 심리적 이득이 큰 작업
- 보통: 중요한 흐름을 조금 전진시키는 일반 작업
- 깊게: 집중과 난이도가 있지만 결과물에 직접 연결되는 작업

깊은 작업은 시간보다 환경과 컨디션이 중요하다. 깊은 작업 준비도가 낮으면 준비 작업을 시키기보다 바로 가벼운 작업으로 낮추는 것이 좋다.

### 추천 이유

AI는 추천 이유를 항상 짧게 보여줘야 한다.

예:

- "현재 `휴대폰 · 산만함 · 가볍게`라서 문서 작성 대신 음성 로그를 추천했어."
- "이번 주 전진 프로젝트이고, 현재 노트북/조용함 환경에서 바로 시작 가능해."
- "최근 로그가 끊겨 있어 복귀용으로 상태 카드 업데이트를 추천했어."

추천 이유가 있어야 사용자가 AI의 판단을 신뢰하거나 수정할 수 있다.

### 추천 화면

실행 추천 화면은 아직 실제로 봐야 판단할 수 있다.

기본 실험안:

- 추천 작업 3개
- 각 작업의 한 줄 이유
- 시작 버튼
- 다른 추천 보기
- 작업 완료 후 로그 입력

3개 추천은 선택 부담을 줄이면서도 AI가 틀렸을 때 대안을 제공하는 중간안이다.

## 7. 상태 전환과 복귀

### 해결하는 문제

유튜브를 보며 멍해진 상태나 회피 상태에서 바로 계획을 보거나 프로젝트를 보면 부담이 커질 수 있다. 먼저 실행 가능한 상태로 돌아오는 입구가 필요하다.

### 5분 유산소 복귀 버튼

가장 현실적인 복귀 행동은 짧은 유산소 운동이다.

예:

- 버피
- 제자리 뛰기
- 짧은 고강도 움직임

목표는 운동 성과가 아니라 멍한 상태를 끊고 몸을 먼저 깨우는 것이다.

복귀 후에는 큰 계획을 보여주기보다, 현재 컨텍스트에 맞는 10분 행동 3개를 보여주는 방향이 덜 부담스럽다.

## 8. 작업 완료 로그 시스템

### 해결하는 문제

프로젝트 맥락이 쌓이려면 작업 완료 후 기록이 필요하다. 하지만 기록이 강제되면 부담이 된다.

### 기본 방식

작업 완료 후에는 기본적으로 로그 입력을 묻되, 스킵 가능해야 한다.

질문:

"방금 한 작업을 한 줄로 남길래?"

입력 방식:

- 텍스트
- 음성
- 스킵

사용자가 두서없이 말하면 AI가 정리한다.

정리 항목:

- 한 일
- 바뀐 상태
- 다음 행동
- 막힌 점

### 스킵 시 자동 로그

사용자가 로그를 스킵하면 AI는 자동 최소 로그를 만든다.

단, 실제 진척을 과장하지 않는다.

좋은 자동 로그:

`자동 기록` - `수집 시스템 정리하기` 작업을 완료로 표시함. 세부 로그 없음.

나쁜 자동 로그:

수집 시스템 설계를 완료함.

### 로그 신뢰도

로그는 신뢰도를 구분한다.

수동/음성 입력 후 AI 정리 로그:

- 신뢰도 높음
- 최근 진척 요약, 프로젝트 상태 업데이트, 다음 행동 갱신, 목표일 조정 근거로 사용

스킵 후 자동 최소 로그:

- 신뢰도 낮음
- 최근 작업 여부에는 반영
- 실제 진척 판단에는 약하게 반영
- 프로젝트 상태를 크게 바꾸는 근거로는 사용하지 않음

## 9. 계획 조정 시스템

### 해결하는 문제

프로젝트 목표일을 직접 정하기 어렵고, 늦어졌을 때 부담만 커지는 문제를 해결한다.

네 프로젝트들은 자기주도적이고 변동성이 크기 때문에 처음 세운 목표일이 틀릴 수 있다. 목표일은 실패 판정이 아니라 현실에 맞게 재계산되는 기준선이어야 한다.

### 작은 결과물과 목표일

프로젝트 전체 마감보다 먼저 "다음 작은 결과물"에 목표일을 붙인다.

예:

- 최종 목표: Work OS 웹 앱 MVP 만들기
- 다음 작은 결과물: 수집 시스템의 입력/분류 흐름 문서화

AI는 목표일 하나를 제안하고 근거를 보여준다. 여러 선택지를 많이 주기보다 하나의 제안이 더 부담이 적다.

### 목표일 재계산

프로젝트가 늦어질 때 AI는 먼저 목표일 조정을 제안한다.

출력:

- 기존 목표일
- 새 목표일 제안
- 왜 조정이 필요한지
- 조정하지 않으면 생길 위험

목표일 변경 시 가장 중요하게 남아야 하는 기록은 "왜 목표일이 바뀌었는지"다.

### 개인화 학습

목표일 변경 이유와 작업 로그가 쌓이면 AI는 다음 패턴을 배워야 한다.

- 내가 결과물을 잘 내는 방식
- 내가 자주 과소평가하거나 과대평가하는 작업 유형
- 환경과 에너지 상태에 따라 가능한 작업
- 생활 패턴이나 작업 환경의 제약

초기에는 정교한 모델보다, 조정 이유를 기억하고 다음 제안에 반영하는 수준이면 충분하다.

## 10. MVP에서 먼저 필요한 것

우선순위가 높은 MVP 요소:

1. 빠른 수집 입력
2. AI 프로젝트 연결
3. 좁은 맥락 질문
4. 프로젝트 목표/현재 상태 카드
5. 태스크 분해
6. 상단 실행 컨텍스트 바
7. 현재 환경과 작업 강도 기반 추천
8. 추천 이유 한 줄 표시
9. 작업 완료 로그
10. 로그 신뢰도 구분

나중으로 미룰 것:

- 화면 녹화 기반 자동 맥락 수집
- 정교한 자동 목표일 예측
- 자동 위치/환경 감지
- 복잡한 태그 체계
- 모든 프로젝트에 대한 완전 자동 요약
- 고급 통계 기반 개인화

## 11. 시스템 간 흐름

전체 흐름은 다음과 같다.

1. 사용자가 해야 할 일이나 생각을 자유 입력한다.
2. AI가 어느 프로젝트와 연결되는지 판단한다.
3. 불확실하면 좁은 질문 하나로 맥락을 회수한다.
4. 프로젝트 이해 시스템이 목표와 현재 상태를 제공한다.
5. AI가 태스크를 쪼갠다.
6. 실행 컨텍스트 바의 환경/강도를 기준으로 지금 가능한 작업으로 변환한다.
7. 실행 추천 화면에서 10분 행동 후보를 보여준다.
8. 사용자가 작업을 수행한다.
9. 완료 후 로그를 남기거나 스킵한다.
10. 로그는 신뢰도에 따라 프로젝트 상태, 최근 진척, 다음 행동, 목표일 조정에 반영된다.
11. 주간 판단 시스템은 프로젝트별 전진/유지/보류/막힘을 근거와 함께 제안한다.
12. 계획 조정 시스템은 작은 결과물의 목표일을 제안하고 필요할 때 재계산한다.

## 12. 아직 실험이 필요한 부분

다음 항목은 대화만으로 확정하기 어렵고, 실제 화면이나 프로토타입을 봐야 판단할 수 있다.

- 실행 추천 화면에서 작업을 몇 개 보여줄지
- 완료 후 로그 입력이 실제로 부담 없는지
- 3개 추천이 적절한지, 1개 집중 추천이 더 나은지
- 프로젝트 상태 카드의 정보 밀도가 적절한지
- AI 질문이 실제로 귀찮지 않은지
- 자동 로그가 시스템 신뢰를 해치지 않는지

## 13. 현재 임시 결론

가장 먼저 설계해야 할 것은 수집 시스템이다.

하지만 수집 시스템은 단순 입력창이 아니다. 프로젝트 이해 시스템과 연결되어야 한다.

핵심 방향:

- 입력은 자유롭게
- 구조화는 AI가
- 중요한 모호함만 질문
- 프로젝트 맥락을 먼저 연결
- 태스크는 현재 환경에서 할 수 있게 변환
- 작업 후 로그는 기본 요청하되 스킵 가능
- 자동 로그는 낮은 신뢰도로 다룸

이 구조가 잡히면 이후 프로젝트 이해, 주간 판단, 실행 추천, 계획 조정을 차례로 구체화할 수 있다.
